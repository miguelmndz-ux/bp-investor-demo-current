The Big Idea

From "6 months" to 2 days

You already have components designed in Figma. What's missing is the code. The Figma → Claude Code → Storybook workflow collapses a 6-month design system buildout into a focused 2-day sprint — by connecting three tools that each do one thing extremely well.

This is not a vague promise. With the Figma MCP Server giving Claude Code direct access to your design files, and the Storybook MCP addon giving Claude Code access to your component library's manifest, you can generate real, production-quality React components from designs, document them in Storybook, and test every state — almost entirely through agentic prompting.

Foundation

What is Storybook & Why It Matters Here

Storybook is an open-source tool that lets you build, document, and test UI components in complete isolation — without needing to run your full app. Think of it as a living design system browser for your codebase.

Concept 01

Stories

A "story" is a single rendered state of a component. A Button might have stories for: Primary, Secondary, Disabled, Loading, With Icon. Stories are what you see in the left nav of Storybook.

Concept 02

Component Isolation

Every component is developed and tested independently. No app context needed. Devs, designers, and PMs can explore the full component surface area without running a build.

Concept 03

Autodocs

Storybook auto-generates documentation from your component code and stories — props tables, usage examples, descriptions. The docs are always in sync with the code.

The Figma parallel: In Figma, you build a Button component once and reuse instances everywhere. Storybook is the code equivalent. You build the component once in React, document every state as a story, and devs reference the Storybook instead of digging through source files. Same mental model, different medium.

Supported Frameworks

Storybook supports React (with Vite or Webpack), Next.js, Vue, Angular, SvelteKit, Svelte, Preact, Web Components, and community frameworks (Nuxt, SolidJS, Analog). For the Figma → Claude Code workflow, React + Tailwind + Vite is the most common and best-supported stack.

End-to-End Process

The Full Workflow

The complete pipeline moves from design to living documented components in four connected stages. Each stage is powered by a different tool, and each handoff is mediated by MCP — eliminating the manual "copy this, paste that" friction of traditional workflows.

01

Design in Figma

Build components with clean naming, variants, and variables. Well-structured Figma = better MCP output.

02

Claude Code reads via Figma MCP

Point Claude at a Figma frame URL + prompt. It calls get_design_context, reads tokens, variants, and layout.

03

Generate React + Stories

Claude generates the component file AND the Storybook stories in one pass. All variant states are captured.

04

Storybook MCP Self-Heals

Storybook MCP lets Claude run interaction + accessibility tests, see failures, and fix its own bugs before you review.

The atoms-first principle: Both the creator transcripts and Figma's docs agree — start with atomic components (buttons, inputs, badges) before composite ones (modals, cards, forms). A modal contains a button. Build the button first. This applies whether you're prompting Claude Code manually or using the full MCP-assisted workflow.
Storybook MCP Server

The Storybook MCP Addon Tools

The @storybook/addon-mcp package runs an MCP server at localhost:6006/mcp alongside your Storybook dev server. This exposes your component library's full context to Claude Code — what components exist, how they're structured, and what tests they need to pass.

⚠️ Note: This addon is currently experimental and under active development. APIs may change.

get-instructionsAlways On

Agent Instructions

Provides Claude Code with standardized instructions for UI component development within your specific project. Returns guidelines so agents follow your conventions when creating components and stories — not generic patterns.

get-story-urlsAlways On

Story URL Lookup

Returns direct URLs to specific stories in your running Storybook. Accepts an absolute path to a story file OR a component/story name. Lets Claude visually verify rendered output and debug render issues without you having to navigate manually.

list-all-componentsManifest

Component Inventory

Returns a complete list of all components in your Storybook. The machine-readable equivalent of browsing your Storybook left nav. Enables Claude to understand the full scope of what already exists before generating anything new.

get-component-detailsManifest

Component Manifest Details

Returns detailed documentation for a specific component: its API (props and types), all variants, design token bindings, and usage examples. This is the token-efficient "curated context" that replaces pointing Claude at your raw source files.

run-interaction-testsSelf-Heal

Interaction Test Runner

Executes the Play function interaction tests defined in your stories. Claude sees pass/fail results and the specific assertion that failed. This closes the autonomous correction loop — Claude generates code, runs tests, and fixes failures without you babysitting it.

run-accessibility-testsSelf-Heal

Accessibility Test Runner

Runs accessibility tests (axe-based) against your stories and returns specific violations. Same self-healing loop as interaction tests — Claude fixes a11y issues autonomously before handing work back to you.

Two packages, one system: @storybook/addon-mcp runs inside your Storybook dev server (port 6006/mcp). @storybook/mcp is a standalone library that can serve component knowledge independently of Storybook — useful for headless or CI contexts. Most teams start with the addon.
Core Capabilities

Major Functionality

These are the headline capabilities that define what makes this workflow fundamentally different from manually building a component library.

🎨→⚛️ Figma to React: Direct Component Generation via MCP
▼

The core value of the workflow: point Claude Code at a Figma frame URL and ask it to generate a React component with Tailwind. The Figma MCP Server reads every style, variant, spacing value, and color token from the design — no manual spec-reading, no copy-pasting hex values.

  • Claude reads Figma variants and maps them directly to React props and Tailwind classes
  • Color tokens, spacing, and typography pulled directly from Figma variables
  • Works for individual components (a single Button) or component systems (all Button variants at once)
  • Specify your tech stack in the prompt: "React + Tailwind, no Shad CN, no additional complexity"
  • Claude also generates the Storybook stories file alongside the component in the same pass
  • Best results when Figma file has clean naming, proper component hierarchy, and well-defined variables
📖 Storybook as the Living Component Library
▼

Every generated component lands in Storybook with stories for every state. Storybook becomes the single source of truth — the place designers, PMs, and devs go to understand what's built, what it looks like, and how to use it.

  • Every variant from Figma becomes a named story — Primary, Secondary, Disabled, Loading, etc.
  • The Controls panel lets anyone interactively toggle props without touching code
  • Autodocs generates prop tables and usage documentation automatically from the component code
  • Devs copy component code directly from the Storybook code panel — no digging through source
  • Built and published Storybook can be shared with the entire team as a static site
  • Supports composition — embed Storybooks from design systems into product Storybooks
🔄 Autonomous Correction Loop: Generate → Test → Self-Heal
▼

This is what separates the Storybook MCP workflow from a simple "generate code" prompt. When the addon is connected, Claude Code doesn't just generate and walk away — it runs tests, sees what fails, and fixes the code autonomously. You step in after the tests pass, not before.

  • Claude generates a component and stories
  • Calls run-interaction-tests to execute Play function tests
  • Reads specific failures and traces them back to prop or logic errors
  • Calls run-accessibility-tests to catch a11y violations
  • Rewrites and re-tests until all tests pass
  • Only hands work back to the developer when the test suite is green
  • Uses get-story-urls to visually inspect rendered output in the browser (especially powerful with Claude in Chrome)
🧠 Component Manifests: Curated Context vs. Raw File Dump
▼

The Component Manifest is the key technical innovation of the Storybook MCP system. Instead of pointing Claude Code at your entire /components directory (token-expensive, noisy, error-prone), the manifest serves exactly what the agent needs — component API, variants, design token bindings, and example usage — in a compact, machine-readable format.

  • Generated automatically by Storybook from your stories, docs, and component code
  • Available as a live endpoint during dev: localhost:PORT/manifests/components.html
  • Built as static JSON files when you run a Storybook production build
  • Dramatically reduces token usage vs. pointing Claude at raw source files
  • Benchmarks show better quality code, faster, with fewer tokens when manifests are present
  • Enables Claude to reuse existing components instead of inventing new ones
🤖 GitHub Issues as Agentic Task Units
▼

For teams using Claude Code with GitHub integration (as shown in the Storybook implementation demo), each component becomes a tightly scoped GitHub Issue. Claude Code picks up the issue, creates a worktree, does the work, commits, and opens a PR — all in one automated flow.

  • Use /issue slash commands to create scoped GitHub Issues from Claude Code
  • Each issue = one component = one PR — keeps review surface small and reviewable
  • Claude creates labels, assigns itself, and updates the project plan document on completion
  • Works in git worktrees so parallel component development is possible
  • Tight scoping is key: "implement storybook button with variants" performs better than vague tasks
  • The developer's role shifts to: write the ticket, review the PR, approve the merge
Supporting Capabilities

Minor Functionality

These features amplify the major capabilities and often make the difference between good and great output quality.

🔌 Claude in Chrome: Visual Verification During Generation
▼

Claude in Chrome is a browser extension that lets Claude Code actually see a running webpage. When generating Storybook components, Claude can launch Storybook at localhost, navigate to the story URL it just generated, visually inspect the rendered output, and catch layout or styling issues that tests alone wouldn't catch.

  • Claude calls get-story-urls to get the direct story URL, then opens it in Chrome
  • Can detect visual mismatches vs. the Figma design reference
  • Useful for catching hover states, responsive behavior, and overflow issues
  • Works alongside (not instead of) interaction and accessibility tests
📋 CLAUDE.md + Design System Rules: Persistent Context
▼

Just like the Figma MCP integration uses create_design_system_rules to generate a persistent context file, your component library workflow benefits from a well-crafted CLAUDE.md that captures your tech stack, naming conventions, and component patterns. The Storybook MCP agent instructions tool supplements this at runtime.

  • Define your stack explicitly: "React + Tailwind only, no Shad CN, no custom CSS"
  • Specify atomic-first order: atoms before molecules before organisms
  • Include design token mapping conventions (e.g., how Figma variable names map to Tailwind classes)
  • Add a project-prompt: "Before any UI work, ALWAYS call the Storybook MCP server for instructions"
  • Reference your PRD so Claude understands what components will be needed across the product
🧪 Testing Suite: Interaction, Accessibility, Visual, Snapshot
▼

Storybook ships with a comprehensive testing suite that integrates directly with the MCP self-healing loop. Each test type catches a different class of error.

  • Interaction tests (Play functions): Simulate clicks, form inputs, keyboard navigation. Claude writes and runs these automatically
  • Accessibility tests: axe-powered checks for WCAG violations. Run autonomously via MCP
  • Visual tests (Chromatic): Pixel-level visual regression. Catches unintended style changes. Requires Chromatic account
  • Snapshot tests: Lightweight checks that component output hasn't changed unexpectedly
  • All tests run in CI — your component library has a test suite from day one
📚 Autodocs: Auto-Generated Component Documentation
▼

When Claude Code generates a component and writes its stories with proper JSDoc/TypeScript annotations, Storybook's Autodocs feature automatically generates a documentation page — prop tables, usage examples, descriptions. Documentation is never out of date because it's derived from the code itself.

  • Prop tables generated from TypeScript types or PropTypes automatically
  • Usage examples pulled directly from stories
  • MDX support for writing richer documentation alongside stories
  • Code panel lets devs copy exact component usage code directly
🎛️ Storybook Essentials: Built-In Development Addons
▼

Storybook ships with a set of "essential" addons pre-installed that make day-to-day component development faster.

  • Controls: Interactive prop toggles in the browser — change variant, size, color in real time
  • Actions: Logs events (clicks, form submissions) fired by your components
  • Backgrounds: Switch between light/dark/custom backgrounds to test component appearance
  • Viewport: Test components at different screen sizes without resizing your window
  • Measure & Outline: Inspect spacing and layout boundaries of rendered components
  • Highlight: Visually highlight specific DOM elements inside a story
📤 Publishing, Sharing & Embedding
▼

Once your component library is built, Storybook can be published and shared — turning your local dev tool into a team-wide reference.

  • Publish to Chromatic (the official Storybook cloud) for a permanent, shareable URL
  • Deploy as a static site to any host (Vercel, Netlify, GitHub Pages)
  • Embed individual stories in Notion, Figma, or Confluence for design reviews
  • Storybook Composition lets you merge multiple Storybooks — useful for monorepos or micro-frontends
  • Figma Design integration addon lets you embed Figma frames next to their code equivalent
Getting Started

Setup Guide

Getting the full Figma → Claude Code → Storybook pipeline running involves three setup layers: Storybook itself, the Figma MCP connection, and the Storybook MCP addon.

1

Install Storybook into your project

Run npm create storybook@latest in your existing project or a new one. The CLI auto-detects your framework (React, Next.js, etc.) and configures everything. Requires Storybook version 9.1.16+ for the MCP addon.

2

Install the Storybook MCP Addon

Run npx storybook add @storybook/addon-mcp — this installs the addon and automatically adds it to your .storybook/main.js. When Storybook is running, the MCP server is available at localhost:6006/mcp.

3

Connect Claude Code to the Storybook MCP Server

Run claude mcp add storybook-mcp --transport http http://localhost:6006/mcp --scope project. Claude Code will auto-detect and connect when you start Storybook. Verify with /mcp list inside Claude Code.

4

Configure the Figma MCP Server

Install and configure the Figma MCP Server (Remote or Desktop mode) as described in the Figma × Claude Code integration. You now have two MCP servers connected: Figma reads designs, Storybook reads components.

5

Write your CLAUDE.md

Define your tech stack, component naming conventions, and the atomic-first build order. Add the system prompt: "Before any UI, frontend or React development, ALWAYS call the Storybook MCP server to get further instructions." This single line dramatically improves output quality.

6

Start small — generate your first atomic component

Pick your most foundational component (usually a Button). In Claude Code: "Generate my Figma selection as a React + Tailwind component, add it to Storybook with stories for every variant, and run the tests." Review the output, check the Storybook UI, iterate.

7

Scale up: work through atoms → molecules → organisms

Once your atoms (Button, Input, Badge, Icon) are done, move to molecules (Form Field, Card, Modal) which are built from atoms. Each component is a tightly scoped task. If using GitHub integration, each one becomes an issue and a PR.

Two MCP servers, one workflow: When both Figma MCP and Storybook MCP are connected, Claude Code has everything it needs in one session — Figma provides the design spec, Storybook provides the component context. Claude knows what already exists (Storybook manifest) and what it should look like (Figma design). That combination is what enables "days, not months."

Mini Learning Path

How to Get Good at This

For someone already strong in Figma and Claude Code. The Storybook part has its own learning curve — but the MCP connection is where the real leverage lives, and that part is fast to learn once Storybook basics are solid.

🚀
Phase 1 — Storybook Orientation 2–4 hrs

Understand Stories, Controls, and Autodocs

Install Storybook on a toy project. Read the official "What's a story?" guide. Manually write 2–3 stories for a component you already have. Explore the Controls panel, the Docs tab, and the Actions panel. Goal: understand the mental model before handing it to Claude.

⚛️
Phase 2 — First Generated Component 2–4 hrs

Generate a Button from Figma, add it to Storybook manually

Use the Figma MCP alone (no Storybook MCP yet). Prompt Claude Code to generate a React + Tailwind Button from a Figma link and add stories. Review what it produced. What was correct? What needed adjustment? Build the muscle of reviewing AI-generated component code.

🤖
Phase 3 — Connect the Storybook MCP 2–3 hrs

Install addon-mcp and watch the self-healing loop work

Install @storybook/addon-mcp and connect Claude Code. Add interaction tests and accessibility tests to a component. Ask Claude to run them and fix failures. Watch the autonomous correction loop in action. This is the moment the workflow "clicks."

🧩
Phase 4 — Build an Atomic System 6–10 hrs

Generate a full set of atoms from a real Figma component library

Take a real Figma file and systematically generate 6–10 atomic components (Button, Input, Badge, Checkbox, Select, Typography, Icon, etc.). Use the Storybook manifest to verify Claude is reusing components as the library grows. This is where CLAUDE.md and atomic ordering pay off.

🏗️
Phase 5 — Molecules and GitHub Integration 5–8 hrs

Build composite components, introduce the Issues/PR workflow

With atoms in place, start generating molecules (Card with Button, Form with Input, Modal with header/body/footer). Each composite component = one GitHub Issue = one PR. Practice the review rhythm: read the code, check the Storybook, approve the merge. Refine your issue-writing to get tighter scopes.

⚡
Phase 6 — Publish, Share, and Maintain Ongoing

Publish your Storybook, establish the update workflow

Publish your Storybook to Chromatic or a static host. Share it with your team. Establish the pattern for how new Figma components get turned into Storybook components going forward. Set up visual regression tests. Integrate Storybook into CI. Now it's a living system, not a one-time sprint.

The compound leverage insight: Well-structured Figma files + a well-built Storybook manifest create a compounding advantage. Every component you add to Storybook makes the manifest richer, which makes Claude Code smarter about reuse, which makes every future component faster to generate and higher quality. The system gets better as it grows.
