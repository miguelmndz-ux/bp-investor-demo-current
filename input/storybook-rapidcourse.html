The Promise — From 6 Months to 2 Days

Building a design system normally takes months. That's not an exaggeration — someone has to manually translate each Figma screen into code, write documentation for every component, write tests so the components don't break, and then keep all of that in sync every time the design changes. That translation work is slow, error-prone, and frankly tedious. This workflow automates the translation step using AI (Claude Code) and connects the tools directly using MCP — Model Context Protocol. Think of MCP as a shared language between apps: instead of you copy-pasting a Figma screenshot into Claude and hoping it guesses right, MCP lets Claude actually read your Figma file and your component library directly, the same way a developer reads source code. The result is that Claude can generate accurate, production-ready components without you acting as a messenger between tools.

Important: "2 days" means 2 focused days on a clean, well-structured Figma file with someone who knows how to prompt Claude Code. Your first run will take longer as you learn. That's what this course is for.

1 Storybook Fundamentals: The Designer's Mental Model ✓ ▼
By the end of this module you will understand every core Storybook concept — stories, CSF3, args, decorators, autodocs, and all 6 built-in Essentials — before any automation touches your project.

Before automating anything, you need to understand what Storybook is doing under the hood. Think of this module as learning to drive before getting on the highway. You'll manually write a few stories yourself — that's the fastest way to truly understand what Claude Code will be generating for you later.

What is a Story?

A story is one specific rendered state of a component, shown in isolation in your browser — completely disconnected from the rest of your app. Think about a Button component: it might have stories for Primary, Secondary, Disabled, Loading, and With Icon. Each of those is a separate story. The Figma analogy makes this click immediately: when you select a Button component in Figma and open the properties panel, you can switch between variants like Type=Primary, Type=Secondary, State=Disabled. A Storybook story is exactly that — one variant, rendered live in a browser, isolated from everything else. Stories live in .stories.tsx files right next to your component file. That isolation is the whole point: you can develop, test, and share a Button without running your entire application. You can hand someone a Storybook URL and they can see every state of every component without touching your codebase.

Component Story Format (CSF3)

CSF3 is just a standard way of writing story files that all of Storybook's tools understand. Like how a Figma file follows a specific format so any Figma feature can read it, your story files follow CSF3 so Storybook's panels, tests, and documentation tools can all read them consistently. It uses plain ES6 module syntax — a default export for the component's metadata and named exports for each individual story. No magic, no custom syntax.

TYPESCRIPT
import type { Meta, StoryObj } from '@storybook/react';
import { Button } from './Button';

// The "meta" (default export) describes the component
const meta = {
  title: 'Components/Button',
  component: Button,
  tags: ['autodocs'], // auto-generate a documentation page
} satisfies Meta<typeof Button>;

export default meta;
type Story = StoryObj<typeof meta>;

// Each named export is one story (one state)
export const Primary: Story = {
  args: { label: 'Click me', variant: 'primary' }
};

export const Disabled: Story = {
  args: { label: 'Click me', variant: 'primary', disabled: true }
};

The meta (default export) is like the component's cover page — it tells Storybook what component this file is about. Each named export (like Primary or Disabled) is a story — one specific state shown in isolation.

Args and ArgTypes — Live-Editable Props

Args are the props you pass to your component in a story. What makes them special is that Storybook's Controls panel lets anyone toggle, type, or select new arg values directly in the browser — no code changes needed. Think of it as the equivalent of Figma's properties panel, but for your code component running live in a browser. You can hand a designer the Storybook URL and they can explore every combination of props themselves. ArgTypes take this further: they let you define how the Controls panel should display each prop — as a color picker, a dropdown, a boolean toggle, or a text field — so the editing experience matches the type of data.

Decorators

Sometimes a component needs context to render correctly — a theme provider, a React Router wrapper, or just some padding to prevent it from touching the edges of the canvas. Decorators are wrappers you can apply to individual stories or to all stories in a file. Think of them like Figma frames: you're not changing the component itself, just giving it a container to sit in. A decorator that wraps every story in a <div style="padding: 2rem"> is one of the most common examples.

Autodocs

Storybook's Autodocs feature reads your component's TypeScript types and your story data and automatically generates a documentation page — complete with a prop table showing every prop, its type, its default value, and a description, plus live usage examples pulled directly from your stories. To enable it, add tags: ['autodocs'] to your meta object and the docs page appears automatically. It's like having a Figma Inspect panel that's always in sync with your code because it's generated from your code. No manual documentation writing required.

The 6 Storybook Essentials

Storybook ships with six built-in "essential" addons pre-installed. These are the panels and tools you'll see in the Storybook UI every day. Here's what each one does:

Essential 1

Controls

Toggle props interactively in the browser. This is your live Figma properties panel for code — change any prop value without touching a file.

Essential 2

Actions

Logs events (clicks, form submissions, any callback) fired by your component as you interact with it. Confirms your event handlers are wired correctly.

Essential 3

Backgrounds

Switch between light, dark, or custom background colors to test how your component looks in different contexts without changing your app.

Essential 4

Viewport

Test your component at preset or custom screen sizes without resizing your browser window. Confirms responsive behavior at every breakpoint.

Essential 5

Measure & Outline

Inspect spacing and layout boundaries of rendered components directly on the canvas. Like Figma's measurement tool — hover to see spacing values.

Essential 6

Highlight

Visually highlight specific DOM elements inside a story. Useful for debugging layout issues and confirming which element corresponds to which code.

MDX — When Autodocs Isn't Enough

MDX is a file format that lets you write richer documentation by mixing Markdown text with live story examples. You'd use it when you want to write a narrative alongside your components — design principles, usage guidelines, accessibility notes, do's and don'ts. For most teams, Autodocs handles the basics automatically and MDX is only used for a handful of complex documentation pages where you need a specific layout or want to tell a story about how a component should be used.

Install Storybook

Storybook has a setup command that auto-detects your framework (React, Next.js, Vue, etc.) and configures everything for you. As of 2025–2026, Storybook is at version 8.x–10.x. Run this in your project root:

BASH
npm create storybook@latest

The CLI will ask a few questions about your setup. For a React + Vite + Tailwind project, accept the defaults. When it's done, you'll have a .storybook/ folder and a src/stories/ folder with example components. Run npm run storybook to verify — it should open in your browser at localhost:6006.

FILE TREE
your-project/
├── .storybook/
│   ├── main.js       ← Storybook configuration
│   └── preview.js    ← Global decorators and parameters
├── src/
│   ├── stories/      ← Example stories (you can delete these)
│   └── components/   ← Your actual components go here
└── package.json

Install the Storybook MCP Addon

The Storybook MCP Addon is what lets Claude Code "read" your component library. Without it, Claude Code has no way to know what components already exist — so it might build a new Button even when you already have one. The addon runs a small server alongside Storybook (at localhost:6006/mcp) that Claude Code can query to get the full manifest of your component library: every component, every prop, every story.

⚠️ Experimental as of early 2026: The @storybook/addon-mcp package is currently experimental. It works, but the API details may change as it matures. Budget extra troubleshooting time and check the Storybook GitHub for the latest install instructions if anything here is outdated.
BASH
npx storybook add @storybook/addon-mcp

That command automatically updates your main.js to include the addon:

JAVASCRIPT
// .storybook/main.js — after running the addon install command
export default {
  addons: [
    '@storybook/addon-essentials',
    '@storybook/addon-mcp', // ← this line gets added automatically
  ],
};

When Storybook is running, the MCP server is live at http://localhost:6006/mcp. You can visit that URL in your browser to confirm it's working — you'll see a JSON response listing your components.

The Two Storybook Packages

Option A

@storybook/addon-mcp

Attaches to your running Storybook dev server. Best for day-to-day development. The one most teams use. Install it with npx storybook add.

Option B

@storybook/mcp

A standalone package that serves component knowledge independently of Storybook. Useful for CI or headless contexts. Most teams only need the addon version.

Figma MCP Server Setup

The Figma MCP Server is what lets Claude Code read your Figma designs. You have two options — both give Claude Code the same capabilities. The choice is about where the server runs, not what it can do.

Mode 1

Remote Mode

A cloud-hosted endpoint. No Figma desktop app required. Works anywhere, including CI pipelines. Set up via the Figma developer settings. Best for most teams.

Mode 2

Desktop Mode

Runs locally via the Figma desktop app. Works offline. Set up via the Figma desktop app's developer settings. Best if you prefer local-only operation.

Follow the setup steps at developers.figma.com/docs/figma-mcp-server for the mode you choose. Once connected, Claude Code can read frames, extract tokens, inspect components, and get code suggestions directly from your Figma file.

Verify the Figma MCP server is already connected. If you haven't done this yet, follow the Figma MCP setup guide at developers.figma.com/docs/figma-mcp-server.

Open Claude Code and run this command to list all connected MCP servers:

Claude knows what already exists (Storybook manifest) and what it should look like (Figma design). This combination is what enables the fast workflow.

3 Writing a CLAUDE.md That Drives Quality ✓ ▼
By the end of this module you will have written a CLAUDE.md that tells Claude Code exactly how to work in your project — achieving Milestone 6.

Claude Code reads a file called CLAUDE.md at the start of every session. Whatever you put in it, Claude remembers for the entire session — your tech stack, your naming conventions, your rules. Without it, Claude makes assumptions. With a well-crafted CLAUDE.md, Claude Code acts like a contractor who already read your project brief before showing up.

Why CLAUDE.md Is Not Optional

Claude Code has no memory between sessions. Without a CLAUDE.md, every new session starts from zero — Claude might generate a component using ShadCN UI even though your project doesn't use it, or build a molecule before any atoms exist, or use class names that don't match your naming convention. These aren't failures of intelligence, they're failures of context. Claude can only follow rules it's been told. The CLAUDE.md is the fix. It's a plain text file at the root of your project that Claude reads automatically at the start of every session — no special command needed. Think of it as the project brief you'd give a new developer on their first day, except Claude reads it every single day.

The Atomic-First Ordering Rule

One of the most important things to put in your CLAUDE.md is the build order. Atomic design means building from small pieces to large ones: atoms (Button, Input, Badge) first, then molecules (Card, Form Field, Alert) that are built from atoms, then organisms (Header, Modal, Sidebar) that are built from molecules. If you don't specify this order explicitly, Claude might start generating a Modal that contains a Button before the Button component exists — leaving you with broken imports and missing dependencies. Include this line verbatim: "Always build atoms before molecules before organisms. Never build a composite component if any of its atomic children are missing." The Figma analogy is direct: you can't use a Button instance inside a Card component in Figma if you haven't created the Button component first. Code works the same way.

Design Token Mapping

If your Figma file uses design tokens (variables for colors, spacing, type scale), you need to tell Claude how those variable names map to your Tailwind classes. For example, your Figma might have a variable called color/primary/500 that maps to bg-blue-500 in Tailwind. Without this mapping in your CLAUDE.md, Claude will use hardcoded hex values instead of your design token classes — making your components inconsistent and harder to update. The mapping table doesn't need to be exhaustive, but it should cover the tokens you use most: your primary colors, your spacing scale, your border radii, and your type scale. Quick tip: You can use the Figma MCP tool create_design_system_rules to auto-generate this section from your actual Figma variables instead of writing it by hand.

The Single Most Important Line

Add this exact line to your CLAUDE.md:

"Before any UI, frontend, or React development, ALWAYS call the Storybook MCP server to get further instructions."

This single instruction makes Claude check your existing component library before generating anything new. Without it, Claude might build a Button when you already have one — wasting tokens and creating duplicates that drift from your established patterns.

Full Annotated CLAUDE.md Example

Here's a complete CLAUDE.md you can use as a starting point. Adapt it to your project. Every section has a comment explaining what it does:

MARKDOWN
# Project: [Your Project Name]

## Tech Stack
- React 18 + TypeScript
- Tailwind CSS v3 (utility classes only — no custom CSS, no inline styles)
- Vite for bundling
- Storybook for component documentation and testing
- NO ShadCN UI, NO Radix, NO MUI — build all components from scratch

## Before Any UI Work
Before any UI, frontend, or React development, ALWAYS call the Storybook MCP
server to get further instructions. This ensures you use existing components
instead of reinventing them.

## Component Build Order (Atomic Design)
Always build in this order. Never skip ahead:
1. Atoms first: Button, Input, Checkbox, Select, Badge, Icon, Typography, Tooltip
2. Molecules second: Card, Form Field, Alert, Modal Header
3. Organisms third: Header, Sidebar, Modal, Form

## Naming Conventions
- Component files: PascalCase (Button.tsx, FormField.tsx)
- Story files: [ComponentName].stories.tsx
- One component per file
- Props interface named [ComponentName]Props

## Design Token Mapping
Figma variable → Tailwind class:
- color/primary/500 → bg-blue-500 / text-blue-500
- color/neutral/100 → bg-gray-100
- spacing/4 → p-4 / gap-4 / m-4
- radius/md → rounded-md
[Add your own token mappings here]

## Component Patterns
- Use Tailwind `cn()` helper for conditional classes
- Prefer composition over prop drilling
- Every component must have TypeScript props interface
- Every component must have a Storybook story file

## Testing
- Write Storybook Play functions for interactive components
- Run accessibility tests via Storybook MCP after each component

Using create_design_system_rules

Run it once after both your Figma variables and your tokens.css are in place — that's when it has everything it needs to generate accurate, project-specific rules.

Returns framework-specific patterns, imports, and conventions for writing stories in your project. This is the first thing Claude calls when starting any UI task — it tells Claude what story format, imports, and naming conventions your project uses.

Dev — Always On

preview-stories

Generates direct preview URLs for stories by story ID or file path. Claude uses this to verify the visual output of a component after generating it — confirming it renders correctly before handing it back to you.

Returns every component and docs entry in your Storybook. This is how Claude checks what already exists before building anything new — without this, Claude might build a Button when you already have one. Unlocks automatically once your first component is written.

Docs — Requires Manifest

get-documentation

Returns full documentation for one specific component: prop types, code snippets, and usage examples. Claude calls this before modifying or composing a component to avoid hallucinating prop names or invalid combinations.

Docs — Requires Manifest

get-documentation-for-story

Returns the detailed documentation for one specific story variant. Use this when you need more usage examples of a single component variant — Claude uses it to understand the exact props and patterns for that variant.

Atomic Design — Atoms, Molecules, Organisms

Atomic design is a way of thinking about components from smallest to largest. An atom is the most basic piece — a Button, an Input, a Badge, a single Icon. A molecule combines atoms — a Form Field is an Input + a Label + an error message. An organism is a full section — a Header with a Logo, a Nav, and some Buttons inside it.

Why it matters here: you can't build a Modal that contains a Button if the Button component doesn't exist yet. Claude Code will try to reuse existing components (via the Storybook manifest) before creating new ones. If your atoms don't exist, Claude either refuses or invents inline components — which is exactly what you don't want.

Level 2

Setting Up Figma for Best MCP Output

The quality of Claude's generated code is directly tied to how well your Figma file is structured. Here's what gets the best results:

CSF3 Deep Dive

CSF3 (Component Story Format 3) is the standard that every Storybook tool relies on. You already saw a basic example in Module 1. Here's a more complete look at the full pattern with TypeScript:

TYPESCRIPT
import type { Meta, StoryObj } from '@storybook/react';
import { Button } from './Button';

// The meta (default export) tells Storybook about the component
const meta = {
  title: 'Components/Button',
  component: Button,
  tags: ['autodocs'],         // generates the Docs tab automatically
  argTypes: {
    variant: {
      control: 'select',
      options: ['primary', 'secondary', 'ghost'],
      description: 'Visual style of the button',
    },
    size: {
      control: 'radio',
      options: ['sm', 'md', 'lg'],
    },
    disabled: { control: 'boolean' },
    onClick: { action: 'clicked' },  // logs click events in Actions panel
  },
} satisfies Meta<typeof Button>;

export default meta;
type Story = StoryObj<typeof meta>;

// Default args shared across all stories
const defaultArgs = {
  label: 'Button',
  variant: 'primary' as const,
  size: 'md' as const,
};

// Named stories — one per Figma variant
export const Primary: Story = { args: { ...defaultArgs } };

export const Secondary: Story = {
  args: { ...defaultArgs, variant: 'secondary' }
};

export const Ghost: Story = {
  args: { ...defaultArgs, variant: 'ghost' }
};

export const Disabled: Story = {
  args: { ...defaultArgs, disabled: true }
};

export const Small: Story = {
  args: { ...defaultArgs, size: 'sm' }
};

export const Large: Story = {
  args: { ...defaultArgs, size: 'lg' }
};

The satisfies Meta<typeof Button> pattern gives you full TypeScript type-checking on your meta — it catches mistyped arg names and invalid control types at compile time, not just at runtime.

The Variant Coverage Principle

Every distinct Figma variant should become a named story — but this does not mean creating a story for every possible combination of variant × state. That leads to a combinatorial explosion that's hard to maintain and adds little value.

Think of it this way: each named story should reveal something visually or behaviorally distinct. If two combinations look and behave the same, one story is enough. The table below shows a sensible coverage for a Button with Primary and Secondary types:

Figma Variant
Named Story
Type=Primary, State=Default
Primary
Type=Secondary, State=Default
Secondary
Type=Primary, State=Disabled
Disabled
Type=Primary, State=Loading
Loading
Type=Primary, Size=Small
Small
Type=Primary, Size=Large
Large
Type=Primary, With Icon
WithIcon
The Controls panel handles the rest. Anyone reviewing your Storybook can already switch between Primary and Secondary, or toggle Disabled, using the controls sidebar — without you needing a named story for every combination. Reserve named stories for states you want to test with a Play function, catch in visual regression (Chromatic), or document explicitly because they're non-obvious.

If Claude misses any variants in the initial generation, you can add them manually by spreading the previous story's args and overriding only what changes.

Story Composition

The ...defaultArgs spread pattern prevents repetition. Instead of re-typing every prop in every story, you define defaults once and only override what's different. This makes the file much easier to read and maintain as the component grows.

Decorators for Context

Some components need a wrapper to render correctly. A full-width hero section needs to be inside a max-width container. A dark-mode component needs a dark background. Add a decorator in the meta or per-story:

TYPESCRIPT
// Per-story decorator
export const DarkMode: Story = {
  args: { ...defaultArgs },
  decorators: [
    (Story) => (
      <div className="bg-gray-900 p-8">
        <Story />
      </div>
    ),
  ],
};

Parameters — Configuring Per-Story Behavior

Parameters control Storybook addons on a per-story basis. You can set a specific background, disable certain controls, or configure how docs behave:

TYPESCRIPT
export const OnDarkBackground: Story = {
  args: { ...defaultArgs },
  parameters: {
    backgrounds: { default: 'dark' },
  },
};

Why Tests Close the Loop

Any AI can write code that looks right. When a test fails, Claude sees the exact assertion that failed and the exact error message. It can then trace that error back to a specific prop or logic mistake and fix it. This loop repeats until all tests pass. You don't babysit it.

Play Functions — The Test Unit

A Play function is an async function you add to a story. It runs automatically after the story renders and simulates user interactions — clicks, typing, keyboard navigation — then asserts that the right things happened. Think of it as a Figma prototype that runs itself and reports a pass or fail.

TYPESCRIPT
import { expect } from '@storybook/test';
import { userEvent, within } from '@storybook/test';

export const Clickable: Story = {
  args: { label: 'Submit', variant: 'primary' },
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);

    // Find the button by its accessible role
    const button = canvas.getByRole('button', { name: 'Submit' });

    // Simulate a click
    await userEvent.click(button);

    // Assert the button is still there and not disabled
    expect(button).not.toBeDisabled();
  },
};

The @storybook/test Toolkit

Everything you need for interaction tests lives in @storybook/test. Here are the four key parts:

Query

within(canvasElement)

Gets the test canvas scoped to your story. Use canvas.getByRole(), canvas.findByText(), canvas.queryBy() to find elements. Like using Figma's layers panel to select a specific layer by name.

Action

userEvent

Simulates real user interactions: userEvent.click(), userEvent.type(), userEvent.keyboard(), userEvent.selectOptions(). All async — use await.

Assert

expect

Checks that elements are in the right state: expect(el).toBeVisible(), expect(el).toBeDisabled(), expect(el).toHaveValue('...').

Organize

step

Groups related interactions for readable output in the Interactions panel: await step('Fill out the form', async () => { ... }).

Accessibility Tests

When @storybook/addon-a11y is installed, the run-story-tests Storybook MCP tool automatically includes accessibility results alongside your interaction test results — both come back in one call. Under the hood it uses axe, an industry-standard accessibility scanner. Axe checks for WCAG violations like missing button labels, low contrast text, missing form field associations, and keyboard trap issues. These are real bugs that affect real users — and Claude fixes them the same way it fixes interaction test failures.

Common violations Claude will catch and fix automatically:

  • "Button has no accessible name" — your button has no text or aria-label. Screen readers can't announce it.
  • "Images must have alternate text" — your <img> has no alt attribute. Screen readers skip it.
  • "Form elements must have labels" — your input has no <label>. Screen readers can't tell users what the field is for.
  • "Color contrast ratio of 2.5:1 is too low" — your text is too light against its background. Users with low vision can't read it.

The Autonomous Self-Healing Loop

  1. 1Claude generates the component (Button.tsx) and stories (Button.stories.tsx), including Play functions
  2. 2Claude calls run-story-tests (Storybook MCP) — the tests run headlessly and return results, including accessibility violations if addon-a11y is installed
  3. 3If tests fail: Claude reads the specific assertion that failed (e.g. "Expected button to be visible but it wasn't found") and traces it back to a prop name mismatch or missing conditional logic
  4. 4Claude rewrites the component, saves the file, and re-runs the tests
  5. 5Claude repeats until all interaction and accessibility checks pass
  6. 6When all tests are green: Claude returns the finished files to you — you step in at this point, not earlier
The key insight: you're not reviewing every intermediate draft. You review the final output after the tests have already confirmed it works.

9 Visual Regression Testing with Chromatic ✓ ▼
By the end of this module, visual regression tests will be set up with Chromatic and your first baseline snapshot set will be captured — achieving Milestone 9.

Interaction tests verify that components behave correctly. Visual regression tests verify that they look correct — catching pixel-level regressions in layout, color, shadows, borders, and spacing that behavior tests miss. Chromatic is the platform purpose-built for this, made by the same team that built Storybook.

What Visual Regression Testing Means

A visual regression test takes a screenshot ("snapshot") of each story at a specific point in time and saves it as the "baseline." Every time you run it again, it compares the new screenshot to the baseline pixel-by-pixel. If anything changed — a border is 1px thinner, a color is slightly different, a font weight changed — it flags it as a change for you to review. You then decide: is this change intentional (accept it, making it the new baseline) or is it a regression (deny it, fix the code)?

Think of it like Figma's version history. You can always go back and compare any current state to a previous saved version. Visual regression does this automatically on every commit.

When Visual Regression Tests Catch What Others Miss

  • A CSS refactor accidentally makes all buttons 1px shorter
  • A font update changes the line height of all body text
  • A Tailwind config change shifts all spacing by 4px
  • A component's shadow or border changes after a dependency update
  • A third-party library update changes the appearance of a select dropdown

    npx chromatic --project-token=YOUR_PROJECT_TOKEN
  5. 5Chromatic builds your Storybook, takes a snapshot of every story, and shows you the results in the Chromatic UI. Since it's your first run, all snapshots are new — you review and "Accept" them to set the baseline.

The Review Flow

Action 1

Accept

The change was intentional. You updated a color, refined spacing, or changed a font. Clicking Accept makes this the new baseline for future comparisons.

Action 2

Deny

The change was a regression. Something broke unexpectedly. Clicking Deny marks the test as failed and blocks the PR until it's fixed.

For each PR, Chromatic runs automatically and reports: "X stories changed." You review each changed story visually before merging. This is your final visual quality gate.

SteadySnap and TurboSnap

Technology 1

SteadySnap

Chromatic's technology for eliminating test flakiness. Before taking a snapshot, it stabilizes animations, freezes dynamic content (like timestamps), and waits for the browser to fully render. Without this, the same story might look slightly different on every run — flaky tests. SteadySnap makes snapshots deterministic.

Technology 2

TurboSnap

Chromatic's speed optimization. Instead of re-snapshotting every story on every commit, TurboSnap analyzes your Git diff and your dependency graph to identify which components could have changed — and only tests those. On large component libraries this can reduce Chromatic run time by 80%+.

Multi-Browser and Viewport Testing

Chromatic runs your snapshots on Chrome, Firefox, Safari, and Edge — in parallel, automatically. You can also configure multiple viewport sizes (mobile, tablet, desktop) and test each story across all combinations. This catches browser-specific rendering differences without you maintaining separate browser test infrastructure.

Chromatic in the PR Workflow

Once Chromatic is connected to GitHub (set up via the Chromatic UI), it adds status checks to every PR:

  • UI Tests — Did any visual regressions appear? Blocks merge if unreviewed changes exist.
  • Storybook Publish — A permanent URL for this specific version of your Storybook. Share it in PR comments for design reviews.
  • UI Review — Separate review step where designers can sign off on visual changes before the PR merges.

GitHub Actions CI Pipeline

CI (Continuous Integration) means your tests run automatically on every PR. No one can merge broken code. Here's the GitHub Actions workflow:

YAML
name: Storybook CI

on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  test-storybook:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '24'  # Node 24+ required for @storybook/addon-mcp

      - name: Install dependencies
        run: npm ci

      - name: Build Storybook
        run: npm run build-storybook

      - name: Run Storybook tests
        run: |
          npx concurrently -k -s first -n "SB,TEST" \
            "npx http-server storybook-static --port 6006 --silent" \
            "npx wait-on tcp:6006 && npx test-storybook"

      - name: Run Chromatic visual tests
        uses: chromaui/action@latest
        with:
          projectToken: ${{ secrets.CHROMATIC_PROJECT_TOKEN }}

What each step does:

  1. 1Checkout code — Gets the latest code from GitHub onto the CI machine
  2. 2Set up Node.js — Installs Node 24 (required for Storybook's MCP tools)
  3. 3Install dependencies — Runs npm ci (like npm install but faster and deterministic in CI)
  4. 4Build Storybook — Compiles your component library to static files
  5. 5Run Storybook tests — Starts a static server for the built Storybook, then runs all Play functions headlessly
  6. 6Run Chromatic — Uploads the build to Chromatic and runs visual regression tests. Requires your CHROMATIC_PROJECT_TOKEN stored as a GitHub secret.

Setting Up the GitHub Secret

  1. 1Copy your Chromatic project token from the Chromatic dashboard
  2. 2Go to your GitHub repo → Settings → Secrets and variables → Actions
  3. 3Click "New repository secret"
  4. 4Name it CHROMATIC_PROJECT_TOKEN, paste the token
  5. 5Click Save — the YAML above can now read it as ${{ secrets.CHROMATIC_PROJECT_TOKEN }}

  • ✓ Build / test-storybook — All interaction and accessibility tests passing

11 Teaching Teammates the Workflow ✓ ▼
By the end of this module you will know how to onboard teammates to the workflow without requiring them to understand MCP internals — achieving Milestone 13.

You don't need everyone on your team to understand how MCP works. What they need to know depends on their role. This module covers the three roles in the workflow, what each role needs to learn, and how to write the team documentation that makes the workflow self-sustaining.

The Three Roles

Role 1

Designer

Structures Figma properly. Clean component naming, variants with properties panel configured, design tokens as Figma Variables. Doesn't need to know how Claude works — but needs to know that their Figma structure is the upstream quality gate for everything that follows.

Role 2

Claude Code Developer

Writes GitHub Issues with Figma links and variant lists, runs Claude Code sessions, reviews PRs from Claude. Needs to understand CLAUDE.md, the generation prompt anatomy, and how to review component code and stories.

Role 3

Reviewer / Consumer

Browses Storybook to find components, copies usage examples from the code panel, reviews PRs for correctness. Doesn't need to write code or use Claude — just needs to know Storybook exists and how to use it as a reference.

What Teammates Don't Need to Know

Most teammates don't need to understand MCP internals. They don't need to know what get_design_context does, how the manifest works, or what a Play function is. What they need is clear documentation of the process they follow. Keep onboarding docs role-specific.

The Figma-First Principle

The single most important thing to teach your design team: clean Figma structure is the upstream quality gate. If a designer names a component layer "Rectangle 42" instead of "Button", Claude reads "Rectangle 42" and generates poor code. If variants are set up as separate frames instead of as Figma Variants, Claude can't extract the variant logic.

Teach designers to:

  • Name all components in PascalCase matching the code name (Button, not button or btn)
  • Use Figma's Variants panel for every component that has multiple states
  • Use Figma Variables (not hardcoded hex values) for all colors and spacing
  • Group components by atomic level in the Figma pages: Atoms, Molecules, Organisms

The Update Cadence

Establish when new Figma components trigger new GitHub Issues. A simple rule: whenever a designer marks a Figma component as "ready for development" (using a Figma status or label), the Claude Code operator creates a new issue. Weekly reviews of the Figma file catch new components. This keeps the code library in sync with the design library.

The Compound Leverage Insight for Your Team

Every component your team adds to Storybook makes the manifest richer. A richer manifest means Claude reuses existing components more reliably, which means every future component takes less time and produces higher-quality output. The system gets smarter as the library grows. Share this with your team so they understand why Storybook maintenance is worth the investment.

Common Pitfalls for New Team Members

  • Vague issue briefs — "Build the card component" is not enough. The issue must include a Figma link and a variant list.
  • Approving PRs without checking Storybook — The code might look fine but render incorrectly. Always open the Storybook link from the PR before approving.
  • Not running the MCP test loop — If Claude generates a component without running tests, there could be hidden accessibility issues or interaction bugs. The CLAUDE.md should enforce this.
  • Skipping Code Connect — Without Code Connect mappings, Claude can't tell which Figma component corresponds to which code component. This leads to duplicate components.
  • Adding atoms after molecules — If a molecule PR arrives before the atom it composes exists, Claude invents the atom inline. Keep the atomic order strict.

Figma feature that maps a Figma component to its code equivalent. Prevents Claude from building duplicates when the mapping is set up.

The compounding benefit where each new component added to Storybook makes the manifest richer, making future generations faster and higher quality.
